# OmniBox-Spider 仓库深度学习笔记：网站类型与推荐写法

更新时间：2026-04-15
学习来源：`https://github.com/Silent1566/OmniBox-Spider/`

## 先说结论

这个仓库最有价值的地方，不是某一个脚本，而是它已经把 OmniBox 的常见业务类型基本铺开了。

我现在得到的最小实战结论是：

## 以后遇到网站，先判断它属于哪一类，再决定写法

而不是一上来就乱写。

当前从仓库结构看，主要类型包括：
- 影视采集站
- 网盘搜索 / 网盘直出
- 解析型站
- 直播站
- 音乐站
- 听书站
- 动漫站
- 短剧聚合站
- Emby 媒体库
- 综合内容站
- 推送类脚本

仓库目录分布也印证了这一点：
- `影视/` 是主战场，数量最多
- `网盘/` 与 `采集/` 是最成熟的两条线
- `模板/` 给的是骨架
- 其他目录给的是不同业务形态的写法样板

## 一、我对这个仓库的整体判断

这个仓库不是“同一种写法复制 100 次”。

它实际上在演示：

## OmniBox 不同业务形态应该怎么写

也就是以后我遇到一个新网站，应该先问：

- 它是采集站吗
- 它是网页 HTML 抓取站吗
- 它是网盘搜索站吗
- 它是需要播放解析吗
- 它是直播吗
- 它是音乐或听书吗
- 它是私有媒体库吗

判断完类型，再选写法。

这就是这次学习最值钱的东西。

## 二、仓库里我提炼出的核心类型

## 1. 模板型
代表：
- `模板/JavaScript/采集站模板.js`
- `模板/JavaScript/推送脚本.js`
- `模板/JavaScript/豆瓣推荐.js`

### 适合场景
- 新源起手
- 骨架搭建
- 明确 OmniBox 的标准结构

### 我学到的写法

#### 采集站模板
适合：
- 已有标准资源站接口
- 接口参数比较稳定
- 不需要自己硬扒 HTML

典型特点：
- 环境变量配置 `SITE_API`
- 统一 `requestSiteAPI()`
- 原始数据统一映射成 `VodItem`
- 把旧式 `vod_play_from + vod_play_url` 转成结构化播放源

### 以后遇到什么站用它
如果网站已经是：
- 苹果 CMS 接口站
- 标准资源站接口
- 提供 `ac=list` / `ac=detail` 这种接口

那就优先走：

## 采集站模板思路

不要去写复杂 HTML 解析。

---

#### 推送脚本模板
适合：
- 用户给一个网盘分享链接
- 源本身不负责搜索
- 重点是 `detail + play`

典型特点：
- 递归遍历网盘目录
- 判断视频文件
- 调 OmniBox 网盘 SDK
- 调刮削接口拿剧集映射

### 以后遇到什么需求用它
如果是：
- 推送链接就能看
- 只要从分享链接生成播放结构
- 不需要首页和分类

那就优先走：

## 推送型写法

只实现最关键的 `detail` 和 `play`。

---

## 2. 影视采集站 / 网页站型
代表：
- `影视/采集/在线之家.js`
- 以及大量 `影视/采集/*.js`

### 适合场景
- 传统影视站
- 有首页、分类、详情、播放页
- 页面结构固定
- 可能有反爬、混淆、加密

### 我学到的写法

#### 这类站的标准套路是
- `axios + cheerio`
- 统一请求 HTML
- 图片 URL 处理
- 必要时解密
- 必要时嗅探真实视频地址
- 网盘线路和普通线路分开处理

`在线之家.js` 特别能说明问题。
它不是只会抓列表，而是已经有：
- HTML 纠偏
- 元数据编码解码
- 图片代理
- 网盘线路识别
- 页面里提取网盘链接
- 嗅探 / 刮削 / 弹幕支持

### 以后遇到什么网站用这种写法
如果网站特点是：
- 页面是 HTML
- 没有稳定公开 JSON 接口
- 播放页要进一步提取
- 可能还混着网盘线路

那就优先走：

## HTML 采集站写法

核心依赖：
- `axios`
- `cheerio`
- 必要时 `OmniBox.sniffVideo()`

### 这类站最重要的不是首页
而是：
- 详情页结构分析
- 播放页真实地址提取
- 图片与请求头处理

---

## 3. 解析型站
代表：
- `影视/解析/movie360.js`

### 适合场景
- 站本身提供的是跳转源
- 视频来自多个平台
- 需要解析接口补全播放

### 我学到的写法

`movie360.js` 很像“平台聚合 + 解析兜底”写法。
它的核心思路是：
- 先拿站点自身列表和详情
- 再把不同站点线路做优先级排序
- 解析接口从环境变量统一配置
- 有默认解析接口兜底
- 分类、详情、搜索结果做缓存

### 以后遇到什么网站用这种写法
如果站点特点是：
- 内容来自腾讯、优酷、爱奇艺等外部平台
- 原站不一定直接给最终直链
- 需要一个或多个解析接口补足

那就优先走：

## 聚合平台 + 解析接口写法

关键点：
- 解析接口不要写死在代码里，优先环境变量
- 线路优先级要可配置
- 缓存要先做

---

## 4. 网盘搜索 / 网盘聚合型
代表：
- `影视/网盘/盘搜.js`
- `影视/网盘/玩偶聚合.js`
- `影视/网盘/聚盘搜索.js`
- 以及多个单盘 / 聚合盘脚本

### 适合场景
- 搜索结果是分享链接
- 详情页本质是网盘目录
- 播放依赖 OmniBox 网盘 SDK

### 我学到的写法

`盘搜.js` 是这类站的典型代表。
核心模式是：
- 搜索请求调用外部盘搜 API
- 返回结果先做平台识别
- 可配置过滤关键词
- 可配置只保留某些网盘类型
- 可接 PanCheck 检测无效链接
- 详情阶段对播放线路按网盘优先级排序
- 大量使用缓存，避免重复解析

### 以后遇到什么网站用这种写法
如果网站特点是：
- 核心价值在搜分享链接
- 结果本身不是直接视频，而是盘链接
- 盘类型很多，质量不一

那就优先走：

## 网盘搜索型写法

核心依赖：
- `OmniBox.getDriveInfoByShareURL`
- `OmniBox.getDriveFileList`
- `OmniBox.getDriveVideoPlayInfo`
- 缓存
- 链接有效性过滤

### 这类源最关键的点
不是页面好不好看，而是：
- 结果过滤
- 无效链接剔除
- 网盘优先级
- 详情缓存

---

## 5. 直播型
代表：
- `直播/虎牙直播.js`
- `直播/斗鱼直播.js`
- `直播/抖音直播.js`

### 适合场景
- 直播平台
- 房间即内容
- 分类页是频道 / 游戏分区
- 详情页是房间信息
- 播放地址实时生成

### 我学到的写法

`虎牙直播.js` 说明这类站和影视站完全不同。
核心模式是：
- `home` 给分区
- `category` 拉房间列表
- `detail` 返回房间信息和可选线路
- `play` 里再动态生成真实播放地址
- 如果主 API 失效，再尝试嗅探

### 以后遇到什么网站用这种写法
如果网站是：
- 直播平台
- 体育直播
- 游戏直播
- 主播房间型内容

那就优先走：

## 直播源写法

重点放在：
- 房间号
- 实时状态
- 真实流地址生成
- 多清晰度或多 CDN 输出

而不是按影视详情页思路硬套。

---

## 6. 音乐型
代表：
- `音乐/枫叶音乐.js`
- `音乐/DJ音乐.js`

### 适合场景
- 音乐搜索
- 歌单
- 单曲播放
- 分类不按“影视片单”，而按歌单/歌曲/标签

### 我学到的写法

`枫叶音乐.js` 很典型。
它的思路是：
- `home` 返回音乐分类
- `category` 里同时处理“歌单列表”和“歌单内歌曲”
- 用 `vod_tag: folder` 表示目录
- 详情非常轻，只负责构建播放源
- `play` 直接返回播放地址

### 以后遇到什么网站用这种写法
如果站点特点是：
- 内容是音频，不是剧集
- 有歌单概念
- 有目录展开概念

那就优先走：

## 音乐源写法

重点是：
- `folder` 目录逻辑
- 搜索结果直接进入播放
- 详情可以很轻

---

## 7. 短剧聚合型
代表：
- `短剧/短剧聚合.js`
- 以及其他短剧目录

### 适合场景
- 多平台 API 聚合
- 每个平台参数、鉴权、结构都不同
- 统一给前端一套短剧输出

### 我学到的写法

`短剧聚合.js` 体现的是：
- 平台配置中心化
- 各平台接口映射统一放配置对象里
- 不同平台 token、加密、参数都要兼容
- 筛选器也要分平台定制

### 以后遇到什么网站用这种写法
如果需求是：
- 多个短剧平台合并成一个源
- 每个平台接口风格都不一样

那就优先走：

## 聚合配置驱动写法

也就是：
- 把平台差异抽进配置表
- 把公共逻辑抽到统一函数
- 不要为每个平台复制一整套脚本

---

## 8. Emby / 私有媒体库型
代表：
- `Emby/模板一.js`
- `Emby/模板二.js`

### 适合场景
- 私有媒体库
- 多账号
- 账号登录
- 目录、媒体、流信息来自媒体服务器 API

### 我学到的写法

这类脚本的核心不是网页抓取，而是：
- 登录
- 鉴权
- 会话信息
- 媒体库遍历
- 媒体流信息构造

`Emby/模板一.js` 明显是 API 客户端思路。
它非常适合“后端服务型站点”，而不是普通网页站。

### 以后遇到什么网站用这种写法
如果目标是：
- Emby
- Jellyfin 类似站
- 私有媒体库系统
- 账号密码登录型媒体 API

那就优先走：

## 客户端 API 写法

核心依赖：
- 账号配置
- 登录态
- token 管理
- 会话请求头

---

## 9. 综合 / 内容平台型
代表：
- `综合/哔哩大全.js`
- 教育、动漫、听书很多也在这一类附近

### 适合场景
- 内容平台不是传统影视站
- 但可以按关键字、分区、合集做内容源
- 有弹幕或附加信息

### 我学到的写法

`哔哩大全.js` 说明：
- 可以把平台内容当资源库来做
- 分类是人为设计的关键词分区
- 搜索、弹幕、封面、时长、集数都需要额外清洗
- Cookie 往往决定能力边界

### 以后遇到什么网站用这种写法
如果站点特点是：
- 本质是内容平台
- 不是标准影视站
- 但内容可以按主题聚合

那就优先走：

## 关键词分区 + 平台接口写法

## 三、我现在形成的网站类型判断法

以后你给我一个网站，我会先按下面顺序判断。

## 第一问：它有没有稳定接口
### 有
优先：
- 接口型写法
- 少碰 HTML

### 没有
优先：
- `axios + cheerio`
- HTML 解析

---

## 第二问：它返回的是视频本体，还是跳板
### 直接视频 / 可拿真实地址
优先：
- 直链播放写法

### 只是跳转页 / 外站页 / 多平台页
优先：
- 解析型 / 嗅探型写法

---

## 第三问：它是单站内容，还是聚合内容
### 单站
优先：
- 直接围绕该站写 `home/category/detail/search/play`

### 聚合
优先：
- 配置驱动
- 平台映射表
- 缓存
- 差异统一清洗

---

## 第四问：它的核心对象是什么
### 影视条目
影视站写法

### 分享链接 / 网盘目录
网盘写法

### 房间
直播写法

### 歌单 / 歌曲
音乐写法

### 私有媒体条目
Emby / 媒体库写法

---

## 四、以后遇到网站时的默认策略

## 1. 标准采集站
用：
- 采集站模板思路

## 2. 普通影视 HTML 站
用：
- `axios + cheerio`
- 必要时嗅探 / 解密

## 3. 盘搜 / 网盘聚合
用：
- 网盘 SDK + 缓存 + 过滤 + 有效性检查

## 4. 聚合跳板站 / 多平台站
用：
- 解析型思路 + 解析接口配置化

## 5. 直播站
用：
- 房间模型 + 动态生成播放流

## 6. 音乐 / 听书站
用：
- `folder` 目录模式 + 轻详情 + 直播放

## 7. 短剧多平台
用：
- 配置表驱动聚合

## 8. Emby / 私有库
用：
- API 客户端思路，不按爬虫页思路写

## 五、这次学习后我得到的最重要经验

## 经验 1
先分类型，再开写。

## 经验 2
不同网站的关键点完全不一样：
- 影视站看详情页和播放页
- 网盘站看链接质量和 SDK
- 直播站看实时流生成
- 音乐站看目录展开
- Emby 看登录和 API

## 经验 3
模板只能做起手，真正成活靠：
- 类型判断
- 数据清洗
- 播放链路处理
- 缓存
- 降级

---

## 六、我后面会怎么用这份学习

以后主人直接给我网站时，我会先回答自己这几个问题：

1. 这是哪一类网站
2. 该优先参考仓库里的哪种写法
3. 最小闭环应该先做哪三个接口
4. 它最容易卡在搜索、详情还是播放
5. 要不要缓存、嗅探、解析、网盘 SDK、Cookie、登录态

这样就不会再用错打法。
