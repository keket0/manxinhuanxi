# 不夜 routes 深入学习笔记（2026-04-15）

学习目录：`/www/manmanai/buye/vod/routes`

## 一、目录分层结论

这个目录不是单一“写一个站源”那么简单，而是一个完整的不夜后端路由体系，至少分成这几层：

1. **影视源层**
   - 目录：`影视/`
   - 典型文件：`bilibili.js`、`哔哩大全融合版.js`、`立播影视.js`
   - 作用：直接提供 `type:4` 的视频接口能力，对外暴露 `/video/xxx` 形式 API。

2. **网盘能力层**
   - 目录：`网盘/`
   - 典型文件：`盘链直登盘检.js`
   - 作用：做盘搜、盘检、自动登录、盘链接状态判断、网盘线路生成。

3. **聚合与配置层**
   - 目录：`配置/`
   - 典型文件：`玩偶聚合.js`、`插件管理.js`
   - 作用：做多站聚合、分组、排序、域名监控、热更新、权限控制、后台管理。

4. **辅助业务层**
   - 目录：`其他/`
   - 典型文件：`全球追更.js`
   - 作用：不是传统站源，而是功能型内容目录、日历追更、跨平台数据编排。

5. **后台接口层**
   - 目录：`API/`
   - 典型文件：`source.js`、`parse.js`
   - 作用：为后台管理页面提供文件列表、语法检查、分类管理、配置读写等能力。

## 二、不夜源的核心写法特征

和 OmniBox 不同，不夜源这里的核心不是 handler 框架抽象，而是：

- 直接导出 `module.exports = async (app, opt) => { ... }`
- 在导出函数里注册路由，例如：
  - `app.get(meta.api, async (req) => { ... })`
  - `app.get('/api/admin/...', ...)`
- 用 `meta` 把这个源挂进站点列表
- 依赖 `opt.sites.push(meta)` 完成注入

也就是说，不夜源本质更像：

**“一个可注册到 Fastify/服务端的路由模块 + 一段站点元信息”**

常见 `meta` 字段：

- `key`
- `name`
- `type`
- `api`
- `searchable`
- `quickSearch`
- `filterable`
- `changeable`
- 额外扩展接口，如 `relayApi`、`configApi`、`loginApi`

## 三、影视源层的几种典型模式

### 1. 接口型单站源
代表：`影视/bilibili.js`

特点：

- 强依赖外部 API
- 结构标准，通常拆成：
  - `_home`
  - `_category`
  - `_detail`
  - `_search`
  - `_play`
- 通过 `req.query` 分流：
  - `play` 走播放
  - `wd` 走搜索
  - `ac=detail` 走分类或详情
  - 默认走首页
- 会自己维护请求头、Cookie、签名、MPD 生成等能力

适用网站：
- B站
- 有稳定 JSON 接口的平台
- 有播放器接口、清晰度接口的平台

### 2. 网页解析型单站源
代表：`影视/立播影视.js`

特点：

- `axios + cheerio` 为主
- 用 `SITE_CONFIG` 统一描述站点规则
- 站点规则集中在：
  - `host`
  - `class_name / class_url`
  - `url / searchUrl`
  - `headers`
  - `filter`
  - `一级 / 二级 / 搜索` 选择器
- 常见辅助函数：
  - URL 修正
  - 选择器解析
  - 属性提取
  - 筛选模板替换
  - HTML 修复
- 会把网页中的“网盘线路”进一步交给 drive 处理

适用网站：
- 苹果 CMS / 海洋 CMS / 首途类模板站
- 站内是 HTML，列表和详情可用选择器提取的网站

### 3. 大型融合增强源
代表：`影视/哔哩大全融合版.js`

特点：

- 文件极大，不只是单站采集，而是“平台能力整合器”
- 一般会包含：
  - 普通视频 + 正片融合
  - 自定义 UP 主
  - 本地配置文件持久化
  - 播放 token / relay / proxy
  - 搜索排序与去噪
  - 兼容线路与主线路双轨
  - 运行时缓存
- 这是“重型专用源”，不是普通模板

适用场景：
- 平台能力复杂
- 需要把多个数据入口融合成一个统一源
- 需要做代理、兼容、缓存、配置持久化

## 四、网盘类源的核心模式

代表：`网盘/盘链直登盘检.js`

这一类不只是“搜资源”，而是完整的业务模块，常见能力：

- 登录态维护
- Cookie 持久化
- 自动登录
- 自动重试
- 盘类型识别
- 盘链接有效性检查
- 屏蔽某些盘类型
- 详情缓存 / 类型缓存 / 页面筛选缓存
- 专门的配置接口和登录接口

典型结构：

- `SITE_CONFIG` 保存主站信息
- `CACHE_TTL` 细分不同数据缓存时长
- `runtimeState` 保存当前运行态，例如 cookie、屏蔽盘类型
- `persistRuntimeConfig()` 持久化运行配置
- `init(server)` 在模块启动时做预热与自动登录

结论：

**不夜的网盘源很多已经不是“采集脚本”，而是“小型后端服务模块”。**

## 五、聚合源的核心模式

代表：`配置/玩偶聚合.js`

这是非常有代表性的不夜聚合写法，关键思想：

1. **站点配置数组化**
   - 每个站点不是分散写死，而是放进 `DEFAULT_SITES`
   - 每个站点包含：
     - `id`
     - `name`
     - `domains`
     - `filterFiles`
     - `categoryUrl`
     - `searchUrl`
     - 选择器
     - 默认分类

2. **域名容灾**
   - 一个站点可以多个域名
   - 定时探测，自动切换可用域名
   - 维护失败计数与健康状态

3. **并发搜索**
   - 多站一起搜
   - 单站设超时
   - 防止个别慢站拖垮整体

4. **聚合详情**
   - 多个站点命中同名资源时，可合并成一个详情
   - 按线路优先级和站点优先级整合线路

5. **筛选外置**
   - 大量筛选条件放在 `筛选/*.json`
   - 主代码只负责加载与应用

结论：

**不夜聚合源的重点不是某个 selector，而是“配置驱动 + 域名容灾 + 并发控制 + 聚合策略”。**

## 六、功能型内容源模式

代表：`其他/全球追更.js`

这类文件说明，不夜 routes 里并不只有“影视站采集”。

它的核心思路是：

- 用外部内容平台 API（这里是 TMDB）生成目录
- 强调“追更、发现、上映日历、目录展示”
- 点击后不一定自己播放，而可能走 `goSearch` 或其他联动能力
- 更像业务功能页，而不是资源站采集页

这类源适合：

- 全球追更
- 排行榜
- 片单
- 日历
- 专题页
- 数据平台映射页

## 七、后台控制器层的写法

代表：`API/source.js`、`API/parse.js`

这些文件说明不夜项目本身自带后台管理思想，常见能力：

- 列出源文件
- 读写配置文件
- 语法检查
- 校验文件安全路径
- 分类配置持久化
- 解析器文件管理

典型特征：

- 严格限制路径穿越
- 通过 `PROJECT_ROOT` 统一定位项目根目录
- 用 `fs.promises` 做文件操作
- 用 `vm.Script` 做 JS 语法检查
- 用 `php -l`、`python -m py_compile` 检查其他脚本语法

这说明“不夜源体系”不仅是前台站源，还包括完整的运维和后台编辑工具链。

## 八、当前提炼出的不夜源写作方法论

以后看不夜源，优先先判断它属于哪一类：

1. **单站接口型**
2. **单站网页解析型**
3. **平台融合型**
4. **网盘业务型**
5. **多站聚合型**
6. **功能目录型**
7. **后台控制器型**

然后再按对应方向去学习，不要混着看。

- 是网页解析，还是接口调用？
- 是否涉及网盘驱动？
- 是否需要多域名容灾？
- 是否需要后台配置接口？
- 是否只是展示目录，不直接播放？

## 九、和 OmniBox 的明确边界

必须继续严格区分：

### OmniBox 关注点
- handler 结构
- context / req 封装
- 统一返回结构
- 开发框架抽象

### 不夜源关注点
- Fastify 路由注册
- `meta + opt.sites.push(meta)` 挂载
- `/video/xxx` 风格接口
- 配置文件外置
- drive 接入
- 业务模块化
- 聚合、容灾、后台管理

一句话：

**OmniBox 更像“框架化源开发”，不夜更像“站点路由模块开发”。**

## 十、影视目录进一步深挖结论

新增抽样文件：`低端影视.js`、`滴答影视.js`、`枫叶.js`、`灵机搜盘.js`

### 1. 不夜影视源内部至少有三种主流写法

#### A. 配置驱动型网页站源
代表：`立播影视.js`、`低端影视.js`、`滴答影视.js`

共性：

- 以 `SITE_CONFIG` 为中心
- 把这些都集中定义：
  - `host`
  - `class_name / class_url`
  - `url / searchUrl`
  - `headers`
  - `filter / filter_def`
  - `推荐 / 一级 / 搜索` 选择器
  - `play_parse / searchable / quickSearch / filterable`
- 下面再配一整套通用工具函数：
  - HTML 修复
  - gzip/base64 解压筛选
  - 模板替换
  - 选择器拆解
  - 属性提取
- 主路由逻辑基本是统一分发：首页 / 分类 / 详情 / 搜索 / 播放

这类写法说明：

**不夜影视源已经形成“半模板化”的站点源工程模式。**

#### B. 专用接口 / 专用平台型
代表：`bilibili.js`、`哔哩大全融合版.js`

共性：

- 不依赖通用 selector 模板
- 直接围绕目标平台 API、播放结构、鉴权细节单独写
- 常带 Cookie、代理、relay、清晰度处理、缓存、用户配置

这类更像“平台插件”，不是普通站源模板。

#### C. 轻量独立脚本型
代表：`枫叶.js`

共性：

- 不走 `SITE_CONFIG + 大量通用工具` 路线
- 直接拆成几个清晰函数：
  - `getClasses`
  - `getHomeRecommend`
  - `getCategoryList`
  - `searchVod`
  - `parsePlayPage`
- 代码更像“按业务点逐个实现”

这类适合结构简单、可快速落地的网站。

### 2. 影视目录里“搜盘”和“普通影视站”已经开始融合

代表：`低端影视.js`、`滴答影视.js`、`灵机搜盘.js`

这里有一个很重要的认识：

- 有的影视源详情里会同时包含普通播放线路和网盘线路
- 有的影视源已经主动砍掉普通直链，只保留网盘驱动结果
- 有的文件虽然放在 `影视/`，但本质已经偏“搜盘影视接口”

比如：

- `低端影视.js` = 普通站源 + 网盘驱动兼容
- `滴答影视.js` = 网页站点外壳 + 只保留网盘线路
- `灵机搜盘.js` = API 驱动型搜盘影视接口，详情阶段直接把盘资源整理成独立线路

所以以后看不夜 `影视/` 目录，不能默认它一定是传统资源站源。

### 3. 不夜影视源的标准写法骨架

如果以后要写一个典型“不夜影视源”，目前最稳的骨架是：

1. 顶部依赖
   - `axios`
   - `cheerio`
   - 视情况加 `crypto-js`、`zlib`

2. 站点主配置
   - `SITE_CONFIG`

3. 通用工具函数
   - URL 拼接
   - 选择器解析
   - 属性提取
   - 模板替换
   - HTML 修复
   - 解密/解压

4. 五大能力函数
   - 首页
   - 分类
   - 详情
   - 搜索
   - 播放

5. 路由注册与 `meta` 输出

## 十一、网盘与配置目录进一步深挖结论

新增抽样文件：`盘搜分组.js`、`盘搜高码-pancheck.js`

### 1. 不夜网盘源有两大方向

#### A. 网盘驱动型
代表：`盘链直登盘检.js`

重点：

- 登录态
- 自动登录
- Cookie 持久化
- 盘类型识别
- 链接有效性校验
- 运行态缓存
- 独立配置接口

#### B. 盘搜聚合型
代表：`盘搜分组.js`、`盘搜高码-pancheck.js`、`灵机搜盘.js`

重点：

- 先调统一盘搜服务接口
- 再把结果按网盘类型聚合或分组
- 再调用独立链接校验服务过滤失效链接
- 再按策略排序输出

这类写法已经是“搜索编排器”，不是单站抓取。

### 2. 盘搜聚合型的稳定流水线

从已读文件看，基本都是这条链路：

1. 请求盘搜服务
   - 如 `/api/search`
2. 收集所有原始链接
3. 去重
4. 调用链接校验服务
   - 如 `links/check`
5. 过滤掉无效链接
6. 按网盘类型 / 关键词 / 时间 / 配置顺序排序
7. 输出成 TVBox 需要的 `vod` 结构
8. 详情阶段交给 `drive.getVod()`

这说明不夜盘搜源最核心的不是“搜”，而是：

**搜后清洗、校验、重排、交给 drive 落播放。**

### 3. 配置目录在不夜体系里的真实作用

`配置/` 目录不是附属品，而是核心能力层。

它主要承担：

- 站点清单配置
- 分组配置
- 排序配置
- 文件分类配置
- 用户分组权限配置
- 盘搜配置
- 聚合源逻辑
- 插件管理
- WebDAV / YouTube 等扩展接入

也就是说：

**不夜很多复杂能力不是写死在单个 js 里，而是“代码 + JSON 配置 + 后台接口”协作。**

### 4. 网盘 / 聚合源的标准写法骨架

如果以后要写一个典型“不夜盘搜/聚合源”，当前最稳骨架是：

1. `init(server)`
   - 拿 `redis`
   - 拿 `log`
   - 拿 `drives`

2. `search()`
   - 调盘搜接口
   - 拉平原始结果
   - 去重
   - 校验链接
   - 排序
   - 组装 `vod` 列表

3. `detail()`
   - 判断链接属于哪个 drive
   - 调 `drive.getVod()`

4. 必要时增加
   - 配置持久化
   - 登录接口
   - runtimeState
   - 缓存 TTL
   - 后台配置读取

## 十二、当前最值得记住的稳定结论

- `影视/` 里不是统一难度，`bilibili.js` 和 `哔哩大全融合版.js` 差别很大，后者属于重型融合源。
- `立播影视.js`、`低端影视.js`、`滴答影视.js` 共同说明：不夜已经形成一套较成熟的“配置驱动型网页影视源”写法。
- `枫叶.js` 说明不夜也允许更轻量的按函数拆分实现，不一定全走大配置模板。
- `灵机搜盘.js` 说明 `影视/` 目录里也可能放偏搜盘、偏 API 编排型的影视接口。
- `盘链直登盘检.js` 代表了“不夜业务型网盘模块”模式。
- `盘搜分组.js`、`盘搜高码-pancheck.js` 代表了“不夜盘搜聚合 + 链接校验 + 排序清洗”模式。
- `玩偶聚合.js` 代表了“不夜多站聚合 + 域名容灾 + 外置筛选”模式。
- `全球追更.js` 说明不夜源也可以是“内容发现目录”，不一定是传统资源站。
- `API/source.js` 与 `API/parse.js` 说明不夜体系自带后台源管理能力。
