# OmniBox JS 写源模式拆解：豆瓣推荐脚本

更新时间：2026-04-15

## 先说结论

这份豆瓣脚本对我最有价值的，不是它抓的是豆瓣，而是它已经展示了一个比较完整的 **OmniBox JavaScript 源写法模式**。

我后面如果要把你给的网站直接写成 JS，可优先复用这里面的这些方法：

1. `home` 里同时返回 `class + list + filters + banner`
2. 用 `context.baseURL` 处理图片代理
3. 用 `context` 而不是硬编码环境上下文
4. 用 `OmniBox.request()` 直接请求站点接口
5. 把源站返回的数据清洗成统一的 `VodItem`
6. 在分类接口里根据不同 `categoryId` 走不同 URL 构造逻辑
7. 错误不直接炸死，优先降级返回空列表

---

## 一、这份脚本属于什么类型

这份脚本只实现了两个接口：
- `home`
- `category`

没有实现：
- `detail`
- `search`
- `play`

说明它更像：

## 推荐 / 分类型源模板
适合：
- 首页推荐
- 分类页
- 筛选页
- 聚合推荐页

不适合直接拿来当“能播”的完整模板。

所以它对我最有价值的地方是：

### 它教的是 OmniBox 的列表层写法
不是播放层写法。

---

## 二、这份脚本体现出的标准 OmniBox JS 结构

## 1. 顶部声明和导出
核心结构是：

- `const OmniBox = require("omnibox_sdk")`
- `const runner = require("spider_runner")`
- `module.exports = { ... }`
- `runner.run(module.exports)`

这就是 OmniBox JS 脚本的标准骨架。

以后我写 JS 源，会默认按这个骨架起手。

---

## 2. 公共小函数抽离
示例里单独抽了：
- `buildDoubanSubjectLink(vodId)`

这说明：

### 凡是会重复出现的 URL 拼接、图片处理、字段清洗，都应该抽函数
以后写别的网站时，我也应该主动拆出：
- `buildDetailLink`
- `normalizeImage`
- `parseSubtitle`
- `buildHeaders`
- `buildSearchUrl`
- `mapItemToVod`

这样后面维护和扩展会轻很多。

---

## 三、这份脚本最值得学的写法模式

## 模式 1：`home` 一次返回四类数据
它的 `home` 不是只给分类，而是同时给：
- `class`
- `list`
- `filters`
- `banner`

这说明 OmniBox 的首页能力其实不弱。

### 对我后面写站的启发
如果目标网站首页信息丰富，我可以主动组织：
- 分类
- 推荐
- 筛选器
- 轮播图

而不是只返回一个光秃秃的列表。

---

## 模式 2：筛选器结构要前置设计
这份脚本把 `filters` 写得很完整。
每个分类单独对应一组筛选器，例如：
- 电影
- 剧集
- 综艺
- 电影筛选
- 电视剧筛选
- 综艺筛选

每个筛选器都遵循统一结构：
- `key`
- `name`
- `init`
- `value`

### 这意味着
以后我写支持筛选的网站时，不应该临时乱拼参数。
应该先做：
- 筛选项设计
- 默认值设计
- 前端显示名和真实参数值映射

这是个很重要的开发顺序。

---

## 模式 3：分类接口按 `categoryId` 分流
这份脚本在 `category()` 里根据不同 `categoryId` 去构造不同 URL：
- `movie`
- `tv`
- `show`
- `movie_filter`
- `tv_filter`
- `show_filter`
- `follow`

### 这是以后写源最实用的模式之一
也就是：

## 一个 `category()` 里可以兼容多套列表逻辑
而不是每个分类单独写一个脚本。

以后我给你写源时，凡是：
- 首页入口不同
- 筛选逻辑不同
- 接口路径不同

都可以优先走这种“按分类分流”的设计。

---

## 模式 4：统一清洗成标准 `VodItem`
这份脚本每次拿到接口结果后，都会统一映射成：
- `vod_id`
- `link`
- `vod_name`
- `vod_pic`
- `type_id`
- `type_name`
- `vod_remarks`
- `vod_year`
- `vod_douban_score`
- `vod_subtitle`
- `search`

### 这是个核心原则
不管源站字段多乱，最后都要整理成 OmniBox 认识的统一结构。

以后我写站，也会优先做一层：

## 站点原始字段 -> OmniBox 标准字段

而不是把站点原始字段直接裸传。

---

## 模式 5：图片统一走代理思路
这份脚本反复用了：
- `context.baseURL`
- `/api/proxy/image?url=...`
- 给图片 URL 附加 `Referer`

### 这非常关键
很多站真正难的不是列表，而是：
- 图片防盗链
- Referer 校验
- 客户端直接加载失败

这份脚本给出的思路是：

## 图片不要想当然直接返回
而是根据情况：
- 直返原图
- 或拼成代理地址
- 或给图片 URL 附带请求头信息

以后我写别的网站，只要图片有防盗链，这个模式可以直接复用。

---

## 模式 6：对错误做降级，而不是整源报废
这份脚本里：
- Banner 获取失败，不影响首页主数据
- 推荐列表失败，也尽量保底返回
- 分类请求失败，则返回空列表

### 这是很成熟的写法
以后写源，我应该尽量做到：
- 非核心模块失败时局部降级
- 核心失败时也返回结构化空结果
- 日志写出来，但不要把整个页面直接打死

这个很重要，因为真实网站经常会：
- 某个接口偶发超时
- 某个模块单独报错
- 某个推荐接口挂了，但搜索还活着

---

## 模式 7：日志只打关键点
它用了：
- `OmniBox.log("warn", ...)`

而不是大量乱打。

### 我后面写源时也应该这样
重点只记录：
- 接口请求失败
- JSON 解析失败
- 关键参数缺失
- 播放地址解析失败

不要把日志写成垃圾堆。

---

## 四、这份脚本暴露出的可复用技巧

## 技巧 1：通过副标题提取年份
它从 `card_subtitle` 里用正则提年份。

这提醒我：

### 很多站没有单独 `year` 字段
但年份可能藏在：
- 标题
- 副标题
- 描述
- 标签串

所以后面做字段标准化时，要多做“从复合文本里抽字段”的处理。

---

## 技巧 2：备注优先用集数信息
它优先从：
- `episodes_info`

提备注。
没有的话，再降级成：
- 新剧 / 新片

### 这说明
`vod_remarks` 不是随便填。
最好优先放：
- 更新到第几集
- 全集
- 画质
- 状态

这是最符合客户端展示价值的字段。

---

## 技巧 3：筛选类接口可以单独造 URL 规则
例如：
- `selected_categories`
- `tags`
- `sort`

这说明很多站的筛选接口不是简单 query 参数堆上去，而是：
- JSON 序列化参数
- 特定标签拼接格式
- 特定编码方式

以后遇到筛选站，我会优先先把它的：
- 筛选参数结构
- 值映射
- 编码规则

摸透，再写。

---

## 技巧 4：收藏 / 追剧这类平台内数据可直接接 SDK
这份脚本有：
- `OmniBox.getSourceCategoryData("follow", page, limit)`

这说明平台内数据不是只能前端自己处理。
脚本也可以直接利用 OmniBox 的源内数据能力。

以后如果你要我做：
- 收藏源
- 历史源
- 追更源

这会很有用。

## 五、这份脚本对我以后“直接看网站写 JS”的真正帮助

## 1. 我更明确了 JS 源该怎么分层
以后写 JS，我会默认拆成：

### 配置层
- 常量
- 请求头
- 基础 URL

### 构造层
- URL 构造函数
- 参数映射函数

### 清洗层
- 图片处理
- 年份提取
- 副标题提取
- 标准备注生成
- 原始 item -> `VodItem`

### 接口层
- `home`
- `category`
- `detail`
- `search`
- `play`

这样结构会比“全部揉在一个函数里”稳得多。

---

## 2. 我会更重视图片、筛选、降级处理
以前很多人写源只盯：
- 能不能拿到列表

但这份脚本提醒我，真正影响实用性的还有：
- 图片是否能正常显示
- 筛选器是否有用
- 某个模块失败时是否能降级

---

## 3. 它证明了 OmniBox 很适合写“接口型数据整理源”
这份脚本不是复杂 HTML 爬虫，而更像：
- 请求接口
- 整理数据
- 返回标准结构

所以以后如果你给我的网站本身就有：
- JSON 接口
- 移动端接口
- 推荐接口
- 搜索接口

那会很好写。

---

## 六、这份脚本的局限，我也记住了

这份脚本不能直接拿来解决：
- 详情页解析
- 播放线路构建
- 视频真实地址提取
- 弹幕
- 搜索

所以它不是完整站模板。

## 它更像“首页+分类+筛选”的优秀参考

这意味着以后我如果要给你写一个能播的完整站，我不能只学它。
还需要补：
- `detail` 模式
- `search` 模式
- `play` 模式
- 防护 / 嗅探 / 播放直链处理

---

## 七、我现在形成的新默认写法

以后主人给我一个网站，我会默认这样干：

## 第一步，先判断网站属于哪类
- 纯接口站
- HTML 站
- 搜索驱动站
- 播放页驱动站
- 聚合接口站

## 第二步，先做公共层
- 基础 URL
- 请求头
- 图片处理
- 字段清洗
- 列表映射器

## 第三步，优先打通核心闭环
- `search`
- `detail`
- `play`

## 第四步，再补列表层体验
- `home`
- `category`
- `filters`
- `banner`

## 第五步，最后做增强
- 缓存
- 弹幕
- 收藏/追更联动
- 按客户端差异返回

---

## 八、最小结论

这份豆瓣脚本我已经吸收成下面这条经验：

## 以后写 OmniBox JS 源，不能只会抓数据，还要同时考虑
- 首页结构
- 分类分流
- 筛选器设计
- 图片代理
- 标准字段清洗
- 出错降级

这些一起做好，源才会像成品，而不是半成品。
